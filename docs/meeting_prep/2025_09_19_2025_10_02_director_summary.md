## Key Decisions Made

### **Decision**: Two-Phase NorthStar Consolidation via Franken Environment
- **Rationale**: Minimize disruption to data engineering teams; allow RBAC/PBAC cleanup without impacting development velocity; front-load transition early in PI to protect delivery schedules
- **Impact**: Development teams transition in first 2 weeks of PI 3.2; Rising Sun repurposed for POC/training only; single NorthStar account by end of PI

### **Decision**: Dual Raw Layer Strategy for Dev Environment
- **Rationale**: Ingestion development requires stable raw layer; data engineering needs fresh production clones for accurate testing
- **Impact**: Created dev_raw_db (ingestion team) and dev_raw_clone_db (engineering team) to prevent clashes; enables parallel workflows

### **Decision**: Split DBT Repositories by Release Cadence
- **Rationale**: OneView near real-time requirements conflict with 4-6 hour analytical batch cycles; need independent orchestration
- **Impact**: Proposed EDP_Streaming, EDP_Data_Integrations, EDP_Data_Domains repos; enables <5 minute latency for OneView

### **Decision**: CSV in S3 as Primary Ingestion (Retire MSK/Kafka)
- **Rationale**: Significant cost reduction; simplified architecture; sufficient performance for near real-time with partitioning
- **Impact**: ARB approved Sept 30; full migration plan in progress; cost savings substantial

### **Decision**: Maintain Raw Layer Control for Abacus Integration
- **Rationale**: Preserve Data Vault 2.0 methodology; maintain security controls (tenant ID); enable AI-powered business rule migration
- **Impact**: Abacus Bronze/Silver flows into our Raw layer (not direct Silver consumption); architectural consistency maintained

### **Decision**: Deferred Merge View Pattern for Near Real-Time
- **Rationale**: Leverages existing Kafka pattern work; lowest implementation risk; proven approach
- **Impact**: UNION ALL view combines ingested + streaming data; dynamic tables apply business rules; <5 min latency achievable
